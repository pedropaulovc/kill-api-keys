Stop passing around API keys.

API keys are shared secrets. They don't expire. They can't be scoped. They leave no audit trail. They are the passwords of the machine world — and we stopped trusting passwords for human authentication years ago. It is time to stop trusting them for machines too.

"But we've always done it this way"

We also used to telnet into production boxes. We used to FTP deploy by dragging files into a folder. We used to email spreadsheets of shared passwords around the office. The industry moved on from every single one of those practices, and nobody misses them.

API keys had their moment. That moment was roughly 2008. Fifteen-plus years later we have OAuth 2.0, OpenID Connect, workload identity, mTLS, and a dozen other standards that don't require you to copy-paste a secret into a .env file and pray nobody commits it.

I am asking you — politely but firmly — to stop using API keys. I have been asking since the first one leaked on GitHub in 2012. Please. I am tired.

The Incident Hall of Horrors

These are not hypothetical risks. These are real breaches, real money lost, and real customer data exposed — all because somebody's API key ended up where it shouldn't have been.

2023 · CircleCI
Every secret, rotated
A breach of CircleCI's platform forced every customer to rotate every secret stored in their CI environment. Thousands of organizations scrambled to replace API keys, tokens, and credentials across their entire infrastructure — overnight.
2022 · Uber
Hardcoded keys on a network share
An attacker social-engineered their way past MFA, then found PowerShell scripts on an internal file share containing hardcoded AWS credentials. Those keys unlocked access to Uber's cloud infrastructure and sensitive data stores.
2022 · Samsung
Keys in the source code
Lapsus$ leaked Samsung's SmartThings source code. Buried in the repo: AWS keys, GitHub tokens, and Artifactory credentials — all in plaintext, all active.
2021 · Codecov
Supply-chain key exfiltration
Attackers modified Codecov's Bash uploader to silently exfiltrate environment variables — including API keys — from the CI pipelines of thousands of companies. Two months went by before anyone noticed.
2021 · Twitch
125 GB of secrets
Twitch's entire source code was leaked, complete with API keys and internal credentials scattered across the codebase. A 125 GB reminder that secrets in source code are only secret until they aren't.
2018 · Tesla
Kubernetes to cryptomining
Attackers found Tesla's Kubernetes console exposed without authentication. Inside: AWS credentials. They used them to run cryptominers on Tesla's cloud bill.
 The scale of the problem: GitHub's secret scanning detects millions of leaked secrets in public repositories every year. API keys are the single most common type. If your authentication strategy depends on nobody accidentally running git push with a secret in the repo, your authentication strategy has already failed.

Why API keys are fundamentally broken

This is not a tooling problem you can solve with secret scanners and pre-commit hooks. API keys are architecturally flawed. They fail on every property that a modern credential should have:

  • No expiration — they live forever unless someone remembers to rotate them. Most don't.
  • No scoping — one key, full access. You can't limit an API key to read-only, to one endpoint, or to one IP range (unless the provider bolted it on as an afterthought).
  • No audit trail — who used this key? When? From where? For what? You'll never know. It's a bearer credential: whoever has the string has the access.
  • Leaked in repos — the #1 type of secret found in public repositories. It's not a matter of if, but when.
  • Shared secrets — both sides store the same secret. Compromise either side and you've compromised both.
  • No standard rotation — every provider has a different rotation mechanism (if any). There is no protocol, no automation, no standard.
  • No revocation granularity — need to revoke access for one consumer? Revoke the key and break every other consumer that shares it.

What to use instead

The good news: better options exist and are widely supported. The right choice depends on your use case:

  • User-facing apps — OAuth 2.0 and OpenID Connect. Delegated access with scoped, time-limited tokens.
  • Machine-to-machine (same cloud) — Workload identity. No secrets at all — the platform proves who you are.
  • Machine-to-machine (cross-boundary) — mTLS, OAuth Client Credentials with private_key_jwt, or SPIFFE.
  • Webhooks — HMAC signatures. The receiver can verify authenticity without storing a bearer credential.

The industry has moved on

This is not a fringe opinion. The biggest cloud providers actively discourage their own API keys:

  • AWS recommends IAM roles and instance profiles over access keys, and flags long-lived keys as a security finding.
  • Google Cloud recommends workload identity federation and service account impersonation over service account keys.
  • Azure recommends managed identities and has built keyless authentication into most of its services.
  • GitHub offers OIDC tokens for Actions workflows and fine-grained tokens with expiration for personal use.

When the providers who issue API keys are telling you not to use them, it might be time to listen.
